第一次部署项目

2026年5月15号是我跟进了一年的项目的交付日期。之前在学校里，无论是课设还是大作业，都是在自己电脑上跑通并得到一个不错的结果，老师就会很满意。因此之前我完全没有接触过实际迁移部署方面的工作，这次第一次做项目交付，确实废了不少力气。

这次项目交付还有一个比较特殊的前提：两台服务器都不能联网。因此核心思路不是在目标服务器上重新安装环境，而是把镜像、代码、数据库、CSV、Python 依赖等内容提前准备好，再整体搬到目标服务器上。换句话说，目标服务器只负责导入、解压、恢复和启动，不能依赖它临时下载任何东西。

一、项目简介

我们做的项目主要是某省的新能源发电功率预测，该项目主要由两部分构成：

  • 前后端 + 数据库：这部分主要由开发组完成，他们最初发给我们的是一个 Docker 镜像。部署到服务器后，可以通过 3306 端口访问数据库，并通过 80 端口查看前端页面。
  • 定时调度预测：这部分统一放在一个固定的项目目录下，每天会定时调度预测函数去获取预测所需的数据、加载模型、得到预测结果，最后写入数据库。

这段 Dockerfile 的核心作用是把“系统底座、Conda 环境、项目代码和启动命令”封装到同一个镜像中。

这份 Dockerfile.python 的核心作用是把基础系统、Conda 环境、项目代码和启动命令封装到同一个 Docker 镜像中。各条命令的含义如下：

  • FROM debian:bullseye-slim：表示使用 Debian 精简版系统作为基础镜像。由于 Python 解释器和依赖包已经包含在打包好的 Conda 环境中，因此这里不需要再使用 python:3.9-slim 这类 Python 基础镜像，也不需要在构建过程中执行 pip install。

  • ENV PYTHONUNBUFFERED=1：表示让 Python 程序的日志实时输出，方便后续通过 docker logs 查看程序运行状态。

  • ENV PYTHONPATH=/home/ubuntu/qinghai/Project：表示设置 Python 的模块搜索路径。项目内部如果有类似 from NewEnergyProject.xxx import xxx 的导入方式，就需要保证 Python 能够找到 NewEnergyProject 的上一级目录。

  • ENV CUDA_VISIBLE_DEVICES=""：表示让容器中的程序默认看不到 GPU，从而强制按 CPU 模式运行。目标服务器没有 CUDA 时，代码中的 torch.cuda.is_available() 会返回 False，程序会自动使用 CPU。

  • COPY conda-env.tar.gz /tmp/conda-env.tar.gz：表示把提前打包好的 Conda 环境压缩包复制到镜像内部的 /tmp 目录下。

  • RUN mkdir -p /opt/conda-env && tar -xzf /tmp/conda-env.tar.gz -C /opt/conda-env && /opt/conda-env/bin/conda-unpack || true：表示在镜像内部创建 /opt/conda-env 目录，将 Conda 环境解压到该目录下，并尝试执行 conda-unpack 修复环境路径。最后的 || true 是为了避免某些情况下 conda-unpack 不存在或执行失败时直接中断镜像构建。

  • COPY NewEnergyProject /home/ubuntu/qinghai/Project/NewEnergyProject：表示把完整的项目目录复制到镜像内部，包括预测脚本、模型文件、配置文件以及项目所需的数据文件。注意，这里的/home/ubuntu/qinghai/Project/NewEnergyProject目录是镜像里的目录。

  • WORKDIR /home/ubuntu/qinghai/Project/NewEnergyProject/project：表示设置容器启动后的默认工作目录。由于 app.py 位于该目录下，因此容器启动后会默认在这个路径中执行后续命令。

  • CMD ["/opt/conda-env/bin/python", "app.py"]：表示容器启动后默认使用 /opt/conda-env/bin/python 运行 app.py。也就是说，目标服务器只需要启动这个镜像，预测程序就会在容器中自动运行，不需要再手动安装 Python 包或配置 Conda 环境。

3.6 启动 Python 预测容器

启动 Python CPU 容器：

这里挂载 data 和 LOG 目录，是为了让目标服务器上的数据文件和日志目录可以与容器内部保持一致，方便后续查看和维护。

  • docker run：表示基于指定镜像创建并启动一个新的容器。

  • -d：表示让容器在后台运行，而不是占用当前终端。启动成功后，Docker 会返回一个容器 ID。

  • --name newenergy-python：表示给这个容器指定名称为 newenergy-python。后续可以直接通过这个名字查看日志、进入容器或停止容器，例如 docker logs newenergy-python。

  • --restart unless-stopped：表示容器异常退出或服务器重启后会自动重新启动，除非手动执行 docker stop 停止它。由于预测程序需要长期运行并每天定时调度，因此这里需要设置自动重启策略。

  • --network host：表示容器使用宿主机的网络。这样容器中的 Python 程序访问数据库或接口时，就可以直接使用宿主机上的网络环境，避免额外配置 Docker 容器网络。

  • -e TZ=Asia/Shanghai：表示设置容器内的时区为 Asia/Shanghai，也就是北京时间。由于项目中存在每天定时调度任务，如果容器时区是 UTC，调度时间可能会和预期相差 8 小时，因此这里需要显式指定时区。

  • -e CUDA_VISIBLE_DEVICES=""：表示让容器中的程序默认看不到 GPU，从而按 CPU 模式运行。目标服务器没有 CUDA 时，程序会自动使用 CPU，不需要修改预测代码。

  • -v /etc/localtime:/etc/localtime:ro：表示将宿主机的时间配置文件挂载到容器内部，并设置为只读。这样容器内部的时间显示可以和宿主机保持一致，避免定时任务因为容器时区不同而提前或延后执行。

  • -v /home/ubuntu/qinghai/Project/NewEnergyProject/data:/home/ubuntu/qinghai/Project/NewEnergyProject/data：表示将宿主机上的 data 目录挂载到容器内部相同路径。这样预测程序读取的数据文件可以直接来自目标服务器本地目录，后续如果需要更新数据文件，也不需要重新构建镜像。

  • -v /home/ubuntu/qinghai/Project/NewEnergyProject/LOG:/home/ubuntu/qinghai/Project/NewEnergyProject/LOG：表示将宿主机上的 LOG 目录挂载到容器内部相同路径。这样容器中程序生成的日志会保存到宿主机目录中，方便在容器外部直接查看和备份。

  • newenergy-python-cpu:latest：表示启动容器所使用的镜像名称。这个镜像中已经包含了 Conda 环境、预测代码和默认启动命令，因此启动后会自动运行 app.py。

四、部署心得

这次项目交付让我第一次真正接触到工程部署和项目迁移。以前在学校做课设或实验时，更多关注的是代码能不能跑通、模型指标是否好看、结果是否符合预期。但真实项目部署需要考虑的问题更多，包括代码是否完整、Python 环境是否一致、数据库是否完整备份、CSV 和模型文件是否遗漏、端口是否正确映射、容器启动后日志是否正常，以及程序能否持续运行并定时写入数据库。

这次我最大的收获是理解了 Docker 在项目迁移中的作用。Docker 并不只是“启动别人给的镜像”，更重要的是可以把项目运行环境、代码和启动命令封装成一个整体，从而减少不同服务器之间环境不一致带来的问题。

对于离线项目交付，最重要的原则是：不要假设目标服务器可以临时下载任何东西，而是要把项目运行所需的一切都提前打包好。只要其中一个环节遗漏，到了目标服务器上就可能因为无法联网而无法补救。

一句话总结：两台机器都离线时，不要在目标服务器安装依赖。要在原服务器或第三台联网机器提前把基础镜像、Python 环境、项目代码、Web 镜像、数据库和 CSV 全部打成离线包；目标服务器只做 docker load、解压、恢复数据库和启动容器。CPU 不需要改 STATION 文件，只要目标服务器没有 CUDA，并且启动 Python 容器时加上 -e CUDA_VISIBLE_DEVICES="" 即可。
